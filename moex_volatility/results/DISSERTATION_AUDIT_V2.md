## 1. Vol-Targeting / Vol-Gating

### 1.1 Точная реализация

**Функция**: `approach_d_v4()` в `scripts/strategies_walkforward_v4.py:891-972`

**Алгоритм** (два варианта):

**Вариант 1 — Standard Vol-Targeting Gate** (строки 917-939):
```python
# scale = (target_vol / sigma_t)^gamma, capped at max_lev
scale = _compute_d_scale_v3(sigma, dp)
gate = (scale >= gate_th).astype(np.float64)   # бинарный: 0 или 1
gated_pos = a_pos * gate                        # {-1, 0, +1} * {0, 1} = {-1, 0, +1}
```

**Вариант 2 — Inverse Vol Gate** (строки 942-959):
```python
# scale = (1/sigma_t) / rolling_mean(1/sigma), capped at max_lev
scale = _compute_d_scale_v3(sigma, dp)          # inverse_vol=True
gate = (scale >= gate_th).astype(np.float64)
gated_pos = a_pos * gate
```

**Функция масштабирования** `_compute_d_scale_v3()` в `scripts/strategies_walkforward.py:2162-2188`:
```
Standard:  scale_t = (V_target / sigma_t)^gamma,  scale_t <= max_lev
Inverse:   scale_t = (1/sigma_t) / SMA_L(1/sigma),  scale_t <= max_lev
```
С опциональным EWM-сглаживанием sigma (span=5 или 20), floor и cap.

### 1.2 Тип: БИНАРНЫЙ VOL-GATE

Это **НЕ** классический vol-targeting. Это бинарный гейт:

| Классический vol-targeting | Наша реализация (vol-gate) |
|---|---|
| w_t = sigma_target / sigma_t | scale_t = (V_target / sigma_t)^gamma |
| w_t — непрерывный, дробный | gate_t = I(scale_t >= threshold) — бинарный {0, 1} |
| Позиция: w_t * signal ∈ R | Позиция: gate_t * A_pos ∈ {-1, 0, +1} |
| Moreira & Muir (2017) | Адаптация для бинарных стратегий |

**Финальные позиции**: строго бинарные {-1, 0, +1}. Подтверждено проверкой в коде (строки 1510-1516):
```python
non_binary = [p for p in unique_pos if abs(p) > 0.01 and abs(abs(p) - 1.0) > 0.01]
# Result: "All positions are binary {-1, 0, +1}"
```

### 1.4 Горизонт sigma_pred

Все три подхода (B, C, D) ищут по h1, h5, h22:
- **Approach B** (`strategies_walkforward.py:151-163`): `horizon ∈ ["h1", "h5", "h22"]` — для sigma-based stop-losses
- **Approach C** (`strategies_walkforward.py:173`): `horizon ∈ ["h1", "h5", "h22"]` — для regime filter (high/low vol)
- **Approach D** (`strategies_walkforward.py:187`): `horizon ∈ ["h1", "h5", "h22"]` — для vol-gate

Лучший горизонт **оптимизируется на валидации** отдельно для каждой стратегии и тестового года.

### 1.5 Обоснование для диссертации

Классический vol-targeting (Moreira & Muir, 2017) предполагает непрерывное масштабирование позиции:
$$w_t = \frac{\sigma_{\text{target}}}{\hat{\sigma}_t}$$

При бинарных позициях ({-1, 0, +1}) дробное масштабирование невозможно. Наша реализация — **естественная адаптация vol-targeting для бинарных стратегий**: вместо плавного масштабирования используется пороговый фильтр (vol-gate). Торгуем когда прогнозируемая волатильность в благоприятном диапазоне, не торгуем когда вне него.

Логика аналогична managed volatility (Barroso & Santa-Clara, 2015): снижаем экспозицию в периоды аномально высокой/низкой волатильности. Пороговая реализация менее параметрична (один параметр gate_threshold вместо непрерывного масштабирования), что снижает риск переобучения.

### 2.3 LSTM/GRU: причины исключения

1. **Хуже бустинга на test 2019**: LSTM H=1 QLIKE 0.3716 vs XGBoost 0.2767 (+34%)
2. **Insufficient data** для walk-forward (`train_neural.log`): нейросетям нужны последовательности (20-30 дней lookback), что сокращает эффективный train при годовых split-ах
3. **Умеренная диверсификация**: корреляция ошибок XGB-LSTM = 0.858, XGB-GRU = 0.804 — ниже чем XGB-HAR (0.94), но выше чем XGB-GARCH (0.72). Недостаточно для оправдания вычислительных затрат.
4. **Вычислительные затраты**: ежегодное переобучение 17 тикеров × 3 горизонта × 10 лет неприемлемо

### 2.4 GARCH: причины исключения из ансамбля

1. **Катастрофический H=1**: QLIKE 1.8017 (vs HAR 0.3052 — в 6 раз хуже)
2. **Природа модели**: GARCH прогнозирует условную дисперсию доходностей, а не реализованную волатильность
3. **Быстрая сходимость**: Multi-step прогнозы быстро сходятся к безусловной дисперсии
4. **Oracle-ценность**: в оракульных блендах GARCH добавляет value (низкая корреляция ~0.5 с ML), но в практической реализации слишком ненадёжен

### 2.6 Устройство гибридной/адаптивной модели

**Hybrid (test 2019)**: Двухкомпонентный бленд
```
pred_t = w_har * HAR-J_t + (1 - w_har) * best_ML_t
```
- w_har оптимизируется на val по Inverse-QLIKE, TSCV, или фиксированно
- best_ML — XGBoost (H=1) или LightGBM (H=5, H=22)

**V1_Adaptive (walk-forward)**: Ежегодный бинарный выбор ML
1. На val-году: сравнить QLIKE XGBoost vs LightGBM
2. Выбрать лучшую ML-модель
3. Оптимизировать w_har на val
4. Применить бленд на тестовом году

**V6_TrimmedW (best H=5)**: Адаптивный тримминг
1. На каждый date-ticker: вычислить ошибку каждой из 3 моделей
2. Отбросить модель с максимальной ошибкой
3. Inverse-deviation-взвесить оставшиеся две

**V7_Rolling_W20 (best H=22)**: Скользящие веса
1. Rolling 20-дневное окно QLIKE-ошибок по тикеру
2. Inverse-QLIKE веса для каждой модели
